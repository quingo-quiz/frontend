# Quingo — UI-план: создание и публикация квизов

> Документ описывает, как мы видим пользовательский интерфейс для работы с квизами
> (этап **создания / редактирования / публикации**, без realtime-игры, каталога и поиска).
> Это основа для следующего шага — проектирования API.
> Дата: 2026-06-17.

Опирается на доменную модель main-service (Quiz / Draft / Snapshot / Card) и на текущую
дизайн-систему фронта: тёмная тема (`background`/`surface`, оранжевый `primary`), скруглённые
карточки, тосты, компоненты `Button`, `Input`, `ConfirmDialog`, модалки, иконки `lucide`.
Все экраны живут внутри защищённой группы `(app)` (требуют авторизации).

---

## 0. Ключевые UX-принципы (из доменной модели)

Прежде чем описывать экраны, зафиксируем смысловые правила, которые UI обязан доносить:

1. **Черновик ≠ опубликованная версия.** Пользователь всегда редактирует **черновик
   (draft)**. Опубликованная версия (**snapshot**) живёт отдельно и не меняется, пока
   пользователь не нажмёт «Опубликовать». UI постоянно должен показывать, в каком состоянии
   квиз и «видят ли это другие прямо сейчас».
2. **Три статуса** (производные, бэкенд считает сам):
   - `UNPUBLISHED` — есть только черновик. «Никто, кроме вас, этого не видит».
   - `PUBLISHED` — есть только снапшот. «Опубликовано, незакоммиченных правок нет».
   - `PUBLISHED_WITH_DRAFT` — есть оба. «Опубликованная версия живёт, а вы готовите новую».
3. **Начать редактировать опубликованный квиз = создать черновик-копию.** Это явное
   действие с понятными последствиями (появляется незакоммиченная версия).
4. **Публиковать можно только валидный квиз:** 1–60 карточек, все карточки корректны,
   непустой заголовок. UI должен заранее показывать, что мешает публикации (чек-лист),
   а не выдавать ошибку постфактум.
5. **Видимость (`PUBLIC`/`PRIVATE`)** влияет только на снапшот. Сделать `PUBLIC` квиз
   без снапшота можно, но «делиться пока нечем» — это надо подсказывать.

## 1. Карта экранов и маршрутов

| Маршрут | Экран | Назначение |
|---|---|---|
| `/quizzes` | **Мои квизы** (список) | Хаб: все квизы пользователя, создание, быстрые действия |
| `/quizzes/[id]/edit` | **Редактор квиза** | Главный экран: редактирование черновика + карточек, публикация |
| `/quizzes/[id]` | **Просмотр квиза** (snapshot) | Предпросмотр опубликованной версии (read-only) |
| — (модалка) | **Новый квиз** | Быстрое создание (title + visibility) |
| — (модалка/панель) | **Редактор карточки** | Часть редактора квиза, не отдельная страница |

Точки входа: пункт **«My Quizzes»** в `Header` (рядом с логотипом/поиском) и/или
домашняя `/`, которую логично превратить из заглушки в дашборд со списком квизов.
Кнопка **«Create Quiz»** доступна и в шапке, и на пустом состоянии списка.

## 2. Экран «Мои квизы» — `/quizzes`

### Назначение
Единая точка, откуда пользователь видит все свои квизы и управляет ими.

### Структура
- **Шапка раздела:** заголовок «My Quizzes», справа — primary-кнопка **`+ Create Quiz`**.
- **Фильтр по статусу** (чипы/табы): `All` · `Drafts` (UNPUBLISHED) · `Published`
  (PUBLISHED + PUBLISHED_WITH_DRAFT). Без поиска/тегов — это вне этапа.
- **Сетка карточек квизов** (адаптивная: 1 колонка на мобиле → 2–3 на десктопе).
- **Empty state:** если квизов нет — иллюстрация/иконка + текст «You don't have any
  quizzes yet» + кнопка «Create your first quiz».
- **Loading state:** скелетоны карточек (как уже сделано на `/settings/security`).

### Карточка квиза в списке
Каждая карточка показывает:
- **Заголовок** квиза. Для `PUBLISHED_WITH_DRAFT` показываем актуальный рабочий заголовок
  (из черновика) — это то, что пользователь правит.
- **Бейдж статуса** с дружелюбными подписями:
  - `UNPUBLISHED` → «Draft» (серый/янтарный).
  - `PUBLISHED` → «Published» (зелёный).
  - `PUBLISHED_WITH_DRAFT` → «Published · editing» (зелёный + оранжевая точка «есть правки»).
- **Бейдж видимости:** иконка-замок «Private» / иконка-глобус «Public».
- **Кол-во карточек** (например, «12 cards»). Считаем по черновику, если он есть, иначе по снапшоту.
- **Отметка времени:** «Edited 2h ago» (modifiedAt черновика) или «Published …» (publishedAt снапшота).
- **Действия:**
  - Главное — **`Edit`** (ведёт в редактор; для `PUBLISHED` сперва инициирует создание черновика, см. §3.1).
  - **`Publish`** — быстрая кнопка, видна для `UNPUBLISHED` и `PUBLISHED_WITH_DRAFT`, **активна только если квиз валиден**; иначе disabled с тултипом-подсказкой.
  - **Overflow-меню (⋯):** `Change visibility` (PUBLIC↔PRIVATE), `View published` (если есть снапшот), `Discard draft` (только PUBLISHED_WITH_DRAFT), `Delete` (destructive).

### Сценарии с этого экрана
- Создать квиз → §2.1.
- Перейти в редактор.
- Сменить видимость (с подтверждением через тост; для приватного без снапшота — предупреждение, что делиться пока нечем).
- Удалить квиз → `ConfirmDialog` (destructive).
- Отменить правки → `ConfirmDialog` (предупреждаем, что черновик-копия будет выброшена).

### 2.1. Модалка «Новый квиз»
По клику `Create Quiz`:
- Поля: **Title** (обязательно, непустой), **Description** (опционально), переключатель
  **Visibility** (Private по умолчанию).
- Кнопка `Create` → создаётся квиз с пустым черновиком (`UNPUBLISHED`) → **редирект в
  редактор** `/quizzes/[id]/edit`, чтобы сразу добавлять карточки.
- *Альтернатива (на выбор при проектировании API):* создавать мгновенно с дефолтным
  заголовком и открывать редактор с inline-редактированием названия. Модалка предпочтительнее —
  меньше «пустых» брошенных квизов.

## 3. Редактор квиза — `/quizzes/[id]/edit`

Сердце всего этапа. **Всегда редактирует черновик (draft).** Предлагаю layout в духе
Kahoot/PowerPoint — «панель слайдов слева + редактор выбранной карточки в центре»,
он лучше всего ложится на «упорядоченный список карточек».

### 3.1. Вход в редактор и состояние черновика
- Квиз `UNPUBLISHED` или `PUBLISHED_WITH_DRAFT` → черновик уже есть, открываем сразу.
- Квиз `PUBLISHED` (черновика нет) → вход в редактор означает **«начать редактирование»**:
  создаётся черновик-копия снапшота, квиз переходит в `PUBLISHED_WITH_DRAFT`.
  - UX: либо открываем редактор и показываем баннер «You're now editing a new version»,
    либо сперва спрашиваем подтверждение. Рекомендую открыть сразу + информирующий баннер
    (меньше трения), а «Discard changes» всегда под рукой.

### 3.2. Верхняя панель (sticky)
- **← Назад** к списку.
- **Заголовок квиза** — редактируется инлайн (клик → input). Непустой.
- **Индикатор статуса/сохранения:**
  - Бейдж статуса (Draft / Published · editing).
  - Индикатор сохранения черновика: «Saved» / «Saving…» (см. §3.6 про модель сохранения).
- **Переключатель видимости** (Private/Public).
- **Кнопки действий:**
  - **`Publish`** (primary) — активна только если квиз валиден; рядом — индикатор готовности
    (например, «5/5 cards valid»). При невозможности — поповер-чеклист «что мешает».
  - **`Discard changes`** — только для `PUBLISHED_WITH_DRAFT` (выбросить черновик, вернуться к снапшоту). С подтверждением.
  - **Overflow (⋯):** `Preview` (предпросмотр черновика), `View published` (открыть снапшот), `Delete quiz` (destructive).

### 3.3. Левая панель — список карточек («слайды»)
- Вертикальный список карточек **по порядку (position с 0)**. Каждый пункт:
  - Номер позиции.
  - Иконка/бейдж типа (Single / Multiple / Text).
  - Короткий превью текста вопроса.
  - **Индикатор валидности** (зелёная точка / красная — с тултипом причины).
  - Выделение активной карточки.
- **Drag-and-drop для переупорядочивания** (перетаскивание меняет `position`).
- Действия на пункте (hover/меню): `Duplicate`, `Delete`.
- Снизу — кнопка **`+ Add card`** с выбором типа (Single choice / Multiple choice / Text input).
- Счётчик «N / 60 cards»; при достижении 60 — `Add card` блокируется с подсказкой.

### 3.4. Центральная область — редактор выбранной карточки
Редактирует одну (активную) карточку. Поля:
- **Тип карточки** — селектор (Single / Multiple / Text). При смене типа, если уже введены
  данные другого типа, **предупреждаем** (опции/тексты могут сброситься).
- **Question text** — многострочное поле, непустой.
- **Timer (seconds)** — число/слайдер, строго > 0 (разумный дефолт, напр. 20с).
- **Данные по типу:**
  - **`SINGLE_CHOICE` / `MULTIPLE_CHOICE`** — список **вариантов (`options`)**:
    - Каждый вариант: input текста (непустой) + переключатель «правильный».
      Для Single — **radio** (ровно один правильный); для Multiple — **checkbox** (≥1).
    - Кнопки `Add option` / удалить вариант. **Минимум 2 варианта**; удаление ниже минимума запрещено.
    - Валидация: Single — ровно 1 правильный; Multiple — ≥1 правильный; нет пустых текстов.
  - **`TEXT_INPUT`** — список **допустимых ответов (`acceptedTexts`)**:
    - Поля строк + `Add answer` / удалить. **Минимум 1 непустой**.
    - Подсказка: «Answers are matched case-insensitively, leading/trailing spaces ignored».
- **Инлайн-валидация** под полями + общий индикатор валидности карточки сверху.

### 3.5. Quiz-level настройки (заголовок, описание, видимость)
- **Title** — инлайн в верхней панели (см. §3.2).
- **Description** — отдельная панель/вкладка «Settings» либо раскрывающийся блок над списком карточек.
- **Visibility** — переключатель в верхней панели; при `PUBLIC` без снапшота — предупреждение.

### 3.6. Модель сохранения черновика (вопрос к API)
Два варианта, влияют на гранулярность эндпоинтов — фиксируем как решение для следующего шага:
- **(A) Автосохранение** изменений черновика (debounce), с индикатором «Saving…/Saved».
  Удобнее для пользователя, но требует частых гранулярных запросов (правка карточки/порядка).
- **(B) Явная кнопка «Save draft»** — реже запросов, но риск потери правок.

Рекомендация: **автосохранение по изменению** (карточка/порядок/мета) — но это закладываем
в дизайн API (нужны покарточные операции, см. §6). На финал согласуем.

### 3.7. Публикация
- Кнопка `Publish` активна, только когда выполнены условия: ≥1 карточка, ≤60, все карточки
  валидны, непустой заголовок.
- По клику — `ConfirmDialog`: «Publish this version? It will become the live version that
  others can play.» (для `PUBLISHED_WITH_DRAFT` — «This will replace the current published version»).
- Успех → квиз становится `PUBLISHED`, черновик «схлопывается» в снапшот → тост «Published».
  Редактор показывает состояние «Published, no pending changes».

## 4. Просмотр опубликованного квиза — `/quizzes/[id]`

Read-only представление **снапшота**.
- Доступен **владельцу** всегда (если снапшот есть); **другим** — только если квиз `PUBLIC`.
- Содержимое: заголовок, описание, список карточек по порядку (тип, вопрос, таймер).
  - Для владельца можно показывать правильные ответы (это его квиз).
  - Это **предпросмотр**, не игра — играть будем позже (realtime вне этапа).
- Кнопки для владельца: `Edit` (→ создание черновика, §3.1), `Change visibility`.
- Если квиз приватный и смотрит не владелец — 403/«not available».

Также из редактора доступен **Preview черновика** (та же раскладка просмотра, но по черновику) —
чтобы автор увидел результат до публикации.

## 5. Сводка валидаций в UI (что подсвечиваем заранее)

| Уровень | Правило | Где показываем |
|---|---|---|
| Квиз | `title` непустой | инлайн в шапке редактора |
| Квиз | 1–60 карточек | счётчик + блок `Add`/`Publish` |
| Квиз | все карточки валидны для публикации | чек-лист у кнопки Publish |
| Карточка | `questionText` непустой | под полем |
| Карточка | `timerSeconds` > 0 | под полем |
| Choice | ≥ 2 вариантов, тексты непустые | список опций |
| `SINGLE_CHOICE` | ровно 1 правильный | у переключателей |
| `MULTIPLE_CHOICE` | ≥ 1 правильный | у переключателей |
| `TEXT_INPUT` | ≥ 1 непустой допустимый ответ | список ответов |

Принцип: **валидируем на клиенте до отправки**, но финальную проверку делает бэкенд —
ошибки сервера (`fieldErrors`) показываем тем же механизмом, что и в auth-формах.

## 6. Что из этого UI диктует требования к API (превью следующего шага)

UI выше порождает следующие потребности (детально проработаем отдельно):
- **Список квизов** — лёгкий ответ: `id`, рабочий `title`, `status`, `visibility`,
  число карточек, отметка времени. Без полного объёма draft/snapshot.
- **Открыть редактор** — полный объект черновика (со всеми карточками и их данными).
- **Создать квиз** — `{ title, description?, visibility }` → возвращает id/объект.
- **Начать редактирование опубликованного** — операция «создать черновик из снапшота».
- **Покарточные операции** (под автосохранение): добавить / изменить / удалить карточку,
  **переставить** (изменить порядок). Решить: целиком черновик vs гранулярно по карточке.
- **Quiz-level правки:** title/description/visibility.
- **Публикация**, **отмена правок (discard draft)**, **удаление квиза**.
- **Просмотр снапшота** (с учётом прав: владелец / PUBLIC).

Конкретные эндпоинты, их гранулярность и контракты опишем на следующем шаге.

## 7. Соответствие текущей дизайн-системе

- Цвета/тема: `background`, `surface`, `primary` (orange) — как в `app.css`.
- Компоненты для переиспользования: `Button`, `Input`, `ConfirmDialog`, паттерн модалок
  (`PasswordModal`/`OtpDisableModal`), `ToastContainer` + `toasts`.
- Состояние: новый rune-контекст для квизов (по аналогии с `userContext`/`securityContext`),
  если понадобится глобально; локальное состояние редактора — через `$state`/`$derived`.
- Иконки: `lucide-svelte` (напр. `Plus`, `Pencil`, `Trash2`, `Eye`, `Globe`, `Lock`,
  `GripVertical`, `CheckCircle2`, `ListChecks`, `Type`).
- Скелетоны/лоадеры и пустые состояния — в стиле уже существующих экранов.
