# Quingo — контракт API для работы с квизами (main-service)

> Что фронту нужно от **main-service** для этапа создания / редактирования / публикации
> квизов. Описание **функциональное**: с какой сущностью работаем и какое действие
> совершаем; перечислены только поля, которые фронту **точно нужны** (полную модель
> определяет бэкенд). Это вход для фиксации контрактов в TypeSpec → Swagger.
> Дата: 2026-06-17. Опирается на доменную модель и `quiz-ui-plan.md`.

---

## 1. Общие соглашения

- **База:** все эндпоинты под `/api/quizzes`. Фронт ходит только через gateway.
- **Идентичность:** клиент **не передаёт** `ownerId` / `X-User-Id` — gateway определяет
  пользователя сам. Все эндпоинты работают «в контексте текущего пользователя».
- **Авторизация:** тем же cookie-механизмом, что и `/api/auth/**`.
- **Конверт ответа:** как в auth-service — успех `{ status, statusMessage, message, data,
  timestamp }`, ошибка `{ status, …, fieldErrors? }`. Поле `data` несёт полезную нагрузку.
- **Ошибки валидации** возвращаются в `fieldErrors` (фронт уже умеет их показывать).

---

## 2. Решения, определяющие форму API

Эти решения из `quiz-ui-plan.md` напрямую диктуют набор эндпоинтов:

1. **Покарточное редактирование + автосохранение.** Редактор сохраняет изменения
   гранулярно (добавить/изменить/удалить/переставить **одну** карточку), а не отправляет
   весь черновик целиком. → нужны отдельные операции над карточками.
2. **При открытии редактора нужен весь черновик** (со всеми карточками и их данными)
   одним запросом.
3. **Список квизов — лёгкий** (без полного объёма карточек): достаточно сводки.
4. **Варианты ответа / допустимые тексты — часть карточки**, отдельных эндпоинтов под
   них не нужно: они приходят и сохраняются вместе с карточкой.

## 3. Эндпоинты по модулям

Ниже — сущность (**Quiz / Draft / Card**), действие и поля, которые точно нужны.

### Модуль: Quiz (квиз целиком)

#### 3.1. Список моих квизов
- **`GET /api/quizzes`**
- **Действие:** получить список квизов текущего пользователя для экрана «My Quizzes».
- **Нужно в ответе (по каждому квизу, лёгкая сводка):**
  - `id`
  - рабочий `title` (из черновика, если есть, иначе из снапшота)
  - `status` (`UNPUBLISHED` | `PUBLISHED` | `PUBLISHED_WITH_DRAFT`)
  - `visibility` (`PUBLIC` | `PRIVATE`)
  - число карточек (`cardCount`)
  - отметка времени для сортировки/подписи (`modifiedAt` черновика или `publishedAt` снапшота)
- **Не нужно:** полный объём карточек, опции, тексты.



#### 3.2. Создать квиз
- **`POST /api/quizzes`**
- **Действие:** создать новый квиз с **пустым черновиком** (результат — `UNPUBLISHED`).
- **Отправляем:** `title` (обязателен), `description?`, `visibility` (дефолт `PRIVATE`) но нужна дать возможность указать.
- **Нужно в ответе:** созданный квиз что сразу перейти в редактор

#### 3.3. Получить квиз целиком (открыть редактор / просмотр)
- **`GET /api/quizzes/{id}`**
- **Действие:** получить полный объект квиза — `status`, `visibility`, `draft?`, `snapshot?`
  со всеми карточками и их типовыми данными (`options` / `acceptedTexts`).
- **Используется:** для редактора (работаем с `draft`) и для просмотра (`snapshot`).
- **Доступ:** владелец видит всё (draft + snapshot); не-владелец — только `snapshot` и
  только если квиз `PUBLIC`, иначе `403/404`.

#### 3.4. Изменить параметры
- **`PATCH /api/quizzes/{id}`**
Я удалил этот блок тк лон был только про видимость, но должна возможность быть менять вообще все - и поисание и заголовок, поэтому эндпоинт должен бьыть более общим переделай. Я хочу чтоб метаданные у снапшота можно было менять, а содержание через драфт сам уже.
#### 3.5. Удалить квиз
- **`DELETE /api/quizzes/{id}`**
- **Действие:** удалить квиз целиком (вместе с черновиком и снапшотом).

### Модуль: Draft (черновик и его жизненный цикл)

#### 3.6. Начать редактирование опубликованного квиза
- **`POST /api/quizzes/{id}/draft`**
- **Действие:** создать **черновик-копию снапшота** (квиз `PUBLISHED` → `PUBLISHED_WITH_DRAFT`).
  Снапшот остаётся нетронутым.
- **Нужно в ответе:** созданный `draft` (чтобы открыть редактор без второго запроса).
- **Условие:** применимо, когда черновиёка ещё нет.

#### 3.7. Изменить мета-данные черновика (название/описание)
- **`PATCH /api/quizzes/{id}/draft`**
- **Действие:** обновить `title` и/или `description` черновика (автосохранение из шапки редактора).
- **Отправляем:** `title?`, `description?`.

#### 3.8. Опубликовать
- **`POST /api/quizzes/{id}/publish`**
- **Действие:** зафиксировать черновик как новый **снапшот** (→ `PUBLISHED`).
- **Серверная валидация (обязательно):** 1–60 карточек, все карточки корректны, непустой
  `title`. При нарушении — ошибка с `fieldErrors` (фронт покажет, что мешает).
- **Нужно в ответе:** обновлённое состояние квиза/`status`.

#### 3.9. Отменить правки (выбросить черновик)
- **`DELETE /api/quizzes/{id}/draft`**
- **Действие:** удалить черновик, оставив прежний снапшот (`PUBLISHED_WITH_DRAFT` → `PUBLISHED`).
- **Условие:** имеет смысл только когда есть и черновик, и снапшот.

---

### Модуль: Card (карточки внутри черновика)

> Все операции — над карточками **черновика** (снапшот неизменяем). Тип-зависимые данные
> (`options` для choice, `acceptedTexts` для text) передаются **внутри карточки** одним телом.

#### 3.10. Добавить карточку
- **`POST /api/quizzes/{id}/draft/cards`**
- **Действие:** добавить новую карточку в черновик.
- **Отправляем:** `type` (`SINGLE_CHOICE` | `MULTIPLE_CHOICE` | `TEXT_INPUT`), `questionText`,
  `timerSeconds`, тип-зависимые данные (`options` или `acceptedTexts`), позиция/место вставки.
- **Нужно в ответе:** созданная карточка с её `id` (и `id` вариантов для choice).

#### 3.11. Изменить карточку
- **`PUT /api/quizzes/{id}/draft/cards/{cardId}`**
- **Действие:** обновить карточку целиком — текст вопроса, таймер, тип и тип-зависимые
  данные (`options` с `isCorrect` / `acceptedTexts`). Автосохранение из редактора карточки.
- **Отправляем:** полное содержимое карточки (включая её варианты/тексты).
- **Замечание:** смена `type` допускается; правила валидности типа проверяет бэкенд.

#### 3.12. Удалить карточку
- **`DELETE /api/quizzes/{id}/draft/cards/{cardId}`**
- **Действие:** удалить карточку из черновика.

#### 3.13. Переставить карточки (изменить порядок)
- **`PUT /api/quizzes/{id}/draft/cards/order`**
- **Действие:** задать новый порядок карточек (drag-and-drop в панели «слайдов»).
- **Отправляем:** список `cardId` в нужном порядке; бэкенд пересчитывает `position`.
- **Почему так:** одним запросом, без рассинхрона позиций при перетаскивании.

## 5. Что НЕ нужно (чтобы не переусложнять)

- **Отдельные эндпоинты под варианты ответа / допустимые тексты** — они часть карточки.
- **Отдельный запрос снапшота** — снапшот приходит в `GET /api/quizzes/{id}`.
- **Эндпоинт «validate draft»** — клиент валидирует до отправки, финальную проверку делает
  `publish`. (Можно добавить позже, если захотим серверную проверку без публикации.)
- **Каталог / лента / поиск / фильтры / теги** публичных чужих квизов — вне этапа.
- **Картинки, баллы, режимы подсчёта, realtime-игра** — вне этапа.
- **Сортировка по дате создания** — поля `createdAt` пока нет.

## 6. Открытые вопросы к бэкенду

1. **Конверт ответа** для main-service подтверждаем тот же, что у auth (`ApiResponse`/`ApiError`)?
2. **Возврат после мутаций:** возвращать обновлённый объект (квиз/карточку) или только
   статус? Фронту удобнее получать обновлённую сущность, чтобы не делать повторный `GET`.
3. **Идемпотентность смены видимости** — `PATCH` с явным значением (а не toggle) — ок?
4. **Создание квиза** — достаточно вернуть `id`, или сразу весь объект с пустым черновиком?
5. **Reorder** — принимаем полный список `cardId`; устраивает ли это бэкенд для пересчёта `position`?
